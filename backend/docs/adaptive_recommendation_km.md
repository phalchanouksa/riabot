## ២.២២.៣. Adaptive Recommendation

Adaptive Recommendation នៅក្នុង backend នេះ គឺជាវិធីសាស្ត្រណែនាំមុខជំនាញដែលមិនចាំបាច់រង់ចាំឱ្យអ្នកប្រើឆ្លើយសំណួរគ្រប់ 256 មុខសិនទេ។ ប្រព័ន្ធអាចធ្វើការប៉ាន់ស្មានលទ្ធផលជាបណ្តោះអាសន្ន ពិនិត្យកម្រិតភាពជឿជាក់ រួចសម្រេចថាគួរសួរសំណួរបន្ថែមឬអត់។ គោលគំនិតសំខាន់គឺសួរឲ្យឆ្លាតជាងមុន និងបញ្ឈប់ឆាប់ជាងមុន នៅពេលប្រព័ន្ធមានទំនុកចិត្តគ្រប់គ្រាន់។

Logic សំខាន់ៗស្ថិតនៅក្នុង `backend/ml_engine/services/adaptive_recommender.py` ហើយត្រូវបាន expose តាម API នៅ `backend/ml_engine/views.py` និង `backend/ml_engine/urls.py`។

### ក. ជំហានដំបូងចាប់ផ្តើមសំណួរ

នៅពេល frontend ចង់ចាប់ផ្តើម adaptive survey វានឹងហៅ `GET /adaptive/start/`។ Backend នឹងហៅ `get_initial_questions()` ហើយត្រឡប់បញ្ជីសំណួរទាំង 256 តាមលំដាប់អាទិភាព។

ចំណុចសំខាន់មួយគឺ ប្រព័ន្ធបច្ចុប្បន្នមិនត្រឹមតែត្រឡប់សំណួរដំបូង 10 ទេ ប៉ុន្តែត្រឡប់បញ្ជីទាំងមូល 256 សំណួរ ដែលបានរៀបលំដាប់រួចជាស្រេច។ វាមានន័យថា phase ដំបូងគឺជាការរៀបលំដាប់សំណួរឱ្យល្អ មុនពេល adaptation ពិតប្រាកដចាប់ផ្តើម។

`get_initial_questions()` ធ្វើការងារដូចខាងក្រោម៖

1. បង្កើត importance weights សម្រាប់សំណួរទាំងអស់
2. បំបែកសំណួរជា 2 ផ្នែក
3. រៀបលំដាប់សំណួរតាមទម្ងន់អាទិភាព
4. សំណួរ interest (`ch1`) ត្រូវបានដាក់មុន
5. សំណួរ skill (`ch2`) ត្រូវបានដាក់បន្ទាប់
6. ផ្នែកខាងលើមួយចំនួនត្រូវបាន randomize បន្តិច ដើម្បីកុំឱ្យលំដាប់មើលទៅដូចម៉ាស៊ីនពេក

លទ្ធផលគឺ frontend អាចចាប់ផ្តើមសួរសំណួរដែលប្រព័ន្ធគិតថាសំខាន់ជាងគេមុន។

### ខ. ការទស្សន៍ទាយជាមួយចម្លើយមិនទាន់ពេញ

នៅពេលអ្នកប្រើឆ្លើយសំណួរបានមួយចំនួន frontend នឹងផ្ញើចម្លើយទៅ `POST /adaptive/predict/`។ Backend នឹងបង្កើត feature vector 256 ធាតុ ដោយសំណួរដែលមិនទាន់ឆ្លើយត្រូវបានដាក់ជា `0` បន្ទាប់មកបញ្ជូនទៅ TabNet ដើម្បីគណនា probability។

បន្ទាប់ពីទទួលលទ្ធផល ប្រព័ន្ធនឹងបង្ហាញ៖

- major ដែលមាន probability ខ្ពស់ជាងគេ
- confidence
- top 3 majors
- uncertainty

### គ. ការសម្រេចថាគួរបន្តសួរឬបញ្ឈប់

ប្រព័ន្ធប្រើ confidence និង uncertainty ដើម្បីសម្រេចថាគួរបន្តសួរឬបញ្ឈប់។ បើចំនួនសំណួរនៅតិចជាង 15 ប្រព័ន្ធនឹងបន្តសួរ។ បើ confidence ខ្ពស់គ្រប់គ្រាន់ ឬ uncertainty ទាបគ្រប់គ្រាន់ បន្ទាប់ពីមានចម្លើយគ្រប់គ្រាន់ហើយ នោះប្រព័ន្ធអាចបញ្ឈប់ និងបង្ហាញលទ្ធផលបាន។

និយាយដោយសង្ខេប ប្រព័ន្ធមិនបញ្ឈប់លឿនពេកទេ ប៉ុន្តែរង់ចាំឱ្យមាន evidence គ្រប់គ្រាន់ជាមុនសិន។

### ឃ. ការជ្រើសសំណួរបន្ទាប់

បើប្រព័ន្ធសម្រេចថាត្រូវបន្ត វានឹងហៅ `get_question_priority()` ដើម្បីជ្រើសសំណួរដែលគួរសួរបន្ទាប់។ Logic សំខាន់គឺ៖

1. ដកសំណួរដែលឆ្លើយរួចចេញ
2. ប្រើ importance weights ដែលបានកំណត់ទុក
3. បន្ថែមអាទិភាពឱ្យសំណួរដែលទាក់ទងនឹង top predicted majors
4. ត្រឡប់សំណួរ 10 មុខបន្ទាប់

ដូច្នេះសំណួរបន្ទាប់មិនមែនជាសំណួរចៃដន្យទេ ប៉ុន្តែជាសំណួរដែលជួយបញ្ជាក់រវាង major ដែលកំពុងនាំមុខ។

### ង. ការពន្យល់លទ្ធផល

នៅពេល frontend ហៅ `POST /adaptive/explain/` ប្រព័ន្ធនឹងបង្កើតសេចក្តីពន្យល់សម្រាប់លទ្ធផល។ វាប្រើ `model.explain()` របស់ TabNet ដើម្បីរកថាសំណួរណាខ្លះដែលមានឥទ្ធិពលខ្លាំងលើការណែនាំ major នោះ។ បន្ទាប់មក backend បម្លែង feature index ទៅជាសំណួរមនុស្សអាចអានបាន ដើម្បីបង្ហាញថា ហេតុអ្វីបានជាប្រព័ន្ធណែនាំ major នេះ។

### សរុប

Adaptive Recommendation គឺជាការរួមបញ្ចូលគ្នារវាង TabNet prediction និង logic សម្រាប់ជ្រើសសំណួរបន្ទាប់ដោយឆ្លាតវៃ។ វាអនុញ្ញាតឱ្យប្រព័ន្ធទស្សន៍ទាយបានមុនពេលឆ្លើយគ្រប់សំណួរ កាត់បន្ថយចំនួនសំណួរដែលមិនចាំបាច់ និងបង្កើនភាពងាយស្រួលសម្រាប់អ្នកប្រើ។
